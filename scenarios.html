Déployer un site Nginx avec HTTPS
Debian/Ubuntu · Nginx · Let's Encrypt — 7 étapes
▼

De zéro à un site servi en HTTPS avec un certificat gratuit qui se renouvelle tout seul. Prérequis : un serveur Debian/Ubuntu et un nom de domaine qui pointe déjà vers son IP.

⌘ commandes Nginx · 🎓 quiz Nginx

  1. Installer Nginx et l'activer au démarrage
    sudo apt update && sudo apt install nginx sudo systemctl enable --now nginx
  2. Vérifier qu'il répond
    curl -I http://localhost

    Tu dois voir HTTP/1.1 200 OK — la page d'accueil par défaut de Nginx.

  3. Créer la racine du site
    sudo mkdir -p /var/www/monsite echo '<h1>Ça marche !</h1>' | sudo tee /var/www/monsite/index.html
  4. Créer le bloc serveur (vhost)
    sudo nano /etc/nginx/sites-available/monsite 
    server {
    listen 80;
    server_name monsite.fr www.monsite.fr;
    root /var/www/monsite;
    index index.html;
}
  5. Activer le site et recharger
    sudo ln -s /etc/nginx/sites-available/monsite /etc/nginx/sites-enabled/ sudo nginx -t sudo systemctl reload nginx

    nginx -t valide la syntaxe AVANT de recharger — toujours le lancer d'abord.

  6. Ouvrir le pare-feu
    sudo ufw allow 'Nginx Full'

    Le profil « Nginx Full » ouvre les ports 80 (HTTP) et 443 (HTTPS).

  7. Passer en HTTPS avec Let's Encrypt
    sudo apt install certbot python3-certbot-nginx sudo certbot --nginx -d monsite.fr -d www.monsite.fr sudo certbot renew --dry-run

    Certbot modifie le vhost tout seul et installe le renouvellement automatique ; le --dry-run vérifie qu'il fonctionnera.

Sécuriser un serveur Linux fraîchement installé
Debian/Ubuntu · SSH · ufw · fail2ban — 7 étapes
▼

Les 20 premières minutes d'un serveur exposé sur Internet : compte admin, connexion par clé, SSH durci, pare-feu et bannissement automatique des attaquants.

⌘ commandes OpenSSL/SSH · 🎓 quiz OpenSSL/SSH

  1. Tout mettre à jour
    sudo apt update && sudo apt full-upgrade
  2. Créer un utilisateur admin (on ne travaille jamais en root)
    sudo adduser tom sudo usermod -aG sudo tom
  3. Générer une clé SSH — sur TON poste, pas sur le serveur
    ssh-keygen -t ed25519 -C "tom@poste" ssh-copy-id tom@ip-du-serveur

    Teste la connexion par clé (ssh tom@ip-du-serveur) avant l'étape suivante, sinon tu risques de t'enfermer dehors.

  4. Durcir la config SSH
    sudo nano /etc/ssh/sshd_config 
    PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
    sudo systemctl restart ssh
  5. Activer le pare-feu (SSH d'abord !)
    sudo ufw default deny incoming sudo ufw allow OpenSSH sudo ufw enable
  6. Bannir les attaques par force brute
    sudo apt install fail2ban sudo systemctl enable --now fail2ban sudo fail2ban-client status sshd
  7. Mises à jour de sécurité automatiques
    sudo apt install unattended-upgrades sudo dpkg-reconfigure -plow unattended-upgrades

Créer un VLAN sur un switch Cisco
Cisco IOS — 7 étapes
▼

Segmenter le réseau : créer un VLAN, y affecter des ports, monter un trunk vers un autre switch, puis sauvegarder pour que tout survive au redémarrage.

⌘ commandes Cisco IOS · 🎓 quiz Cisco IOS

  1. Passer en mode configuration
    enable configure terminal
  2. Créer le VLAN et le nommer
    vlan 10 name BUREAU exit
  3. Affecter un port au VLAN
    interface fastEthernet 0/1 switchport mode access switchport access vlan 10 exit
  4. Ou toute une plage de ports d'un coup
    interface range fastEthernet 0/2 - 12 switchport mode access switchport access vlan 10 exit
  5. Monter le trunk vers l'autre switch
    interface gigabitEthernet 0/1 switchport mode trunk switchport trunk allowed vlan 10,20 end

    Le trunk transporte plusieurs VLAN sur un seul lien ; allowed vlan limite lesquels.

  6. Vérifier
    show vlan brief show interfaces trunk
  7. Sauvegarder — sinon tout disparaît au reboot
    copy running-config startup-config

Créer puis sauvegarder une VM sur Proxmox
Proxmox VE · qm · vzdump — 7 étapes
▼

Tout en ligne de commande : créer une VM Debian, la démarrer, puis la sauvegarder à chaud et savoir la restaurer.

⌘ commandes Proxmox · 🎓 quiz Proxmox

  1. Repérer le stockage et les ISO disponibles
    pvesm status ls /var/lib/vz/template/iso
  2. Créer la VM (id 100)
    qm create 100 --name debian-vm --memory 2048 --cores 2 --net0 virtio,bridge=vmbr0
  3. Ajouter le disque et l'ISO d'installation
    qm set 100 --scsi0 local-lvm:32 qm set 100 --cdrom local:iso/debian-12.iso qm set 100 --boot order='scsi0;ide2'
  4. Démarrer et suivre
    qm start 100 qm status 100

    L'installation se termine dans la console de l'interface web (ou qm terminal 100 si la VM a un port série).

  5. Lister les VM du nœud
    qm list
  6. Sauvegarder à chaud (snapshot)
    vzdump 100 --storage local --mode snapshot --compress zstd

    Le mode snapshot sauvegarde sans arrêter la VM.

  7. Restaurer (vers un nouvel id 101)
    qmrestore /var/lib/vz/dump/vzdump-qemu-100-*.vma.zst 101

Conteneuriser et publier une application avec Docker
Docker · Docker Hub — 7 étapes
▼

Du code source à l'image publiée sur Docker Hub : construire, lancer, déboguer, publier, nettoyer.

⌘ commandes Docker · 🎓 quiz Docker

  1. Écrire le Dockerfile à la racine du projet
    FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]
  2. Construire l'image
    docker build -t mon-app:1.0 .
  3. Lancer un conteneur
    docker run -d -p 8080:3000 --name app mon-app:1.0

    -p 8080:3000 : le port 3000 du conteneur devient le 8080 de ta machine.

  4. Vérifier que ça tourne
    docker ps docker logs -f app curl http://localhost:8080
  5. Entrer dans le conteneur pour déboguer
    docker exec -it app sh
  6. Publier sur Docker Hub
    docker login docker tag mon-app:1.0 monuser/mon-app:1.0 docker push monuser/mon-app:1.0
  7. Nettoyer
    docker stop app && docker rm app docker image prune

Installer MySQL et automatiser les sauvegardes
MySQL/MariaDB · cron — 6 étapes
▼

Base de données prête pour une appli : installation sécurisée, utilisateur dédié aux droits limités, et un dump compressé chaque nuit à 3 h.

⌘ commandes MySQL · 🎓 quiz MySQL

  1. Installer et sécuriser
    sudo apt install mysql-server sudo mysql_secure_installation
  2. Créer la base et l'utilisateur applicatif
    sudo mysql 
    CREATE DATABASE boutique;
CREATE USER 'app'@'localhost' IDENTIFIED BY 'MotDePasseFort!42';
GRANT ALL PRIVILEGES ON boutique.* TO 'app'@'localhost';
FLUSH PRIVILEGES;

    L'appli n'a de droits QUE sur sa base — jamais root dans le code.

  3. Tester la connexion
    mysql -u app -p boutique
  4. Faire un dump manuel
    mysqldump -u root -p boutique > boutique-$(date +%F).sql
  5. Automatiser chaque nuit à 3 h avec cron
    sudo crontab -e 
    0 3 * * * mysqldump boutique | gzip > /var/backups/boutique-$(date +\%F).sql.gz

    Dans un crontab, le % doit être échappé en \%. Pense à créer /var/backups et à donner l'accès root via ~/.my.cnf.

  6. Restaurer un dump
    gunzip < /var/backups/boutique-2026-07-03.sql.gz | mysql -u root -p boutique

Diagnostiquer une panne réseau, étape par étape
Linux · ip · dig · tcpdump — 7 étapes
▼

« Internet ne marche plus » : la méthode en couches, de la carte réseau jusqu'à la capture de paquets, pour localiser la panne au lieu de deviner.

⌘ commandes universelles · ⌘ tcpdump · 🎓 quiz

  1. La machine a-t-elle une IP et une route ?
    ip a ip route

    Pas d'IP → problème DHCP/câble. Pas de default via … → pas de passerelle.

  2. La passerelle répond-elle ?
    ping -c 4 192.168.1.1

    KO → problème local (câble, Wi-Fi, box). OK → on continue vers l'extérieur.

  3. Internet répond-il en IP pure ?
    ping -c 4 8.8.8.8

    Si l'IP répond mais pas les sites web → c'est le DNS, pas la connexion.

  4. Le DNS résout-il ?
    dig monsite.fr resolvectl status
  5. Le service distant écoute-t-il ?
    nc -zv monsite.fr 443 ss -tulnp

    nc -zv teste un port distant ; ss -tulnp liste ce qui écoute en local.

  6. Où ça casse en chemin ?
    traceroute 8.8.8.8
  7. En dernier recours : voir les paquets
    sudo tcpdump -i eth0 -n host 8.8.8.8

    Si les paquets partent sans réponse, la panne est en amont — pas sur ta machine.

Déployer une application sur Kubernetes
Kubernetes · kubectl — 8 étapes
▼

Le cycle de vie complet d'une appli sur un cluster : déployer, exposer, monter en charge, mettre à jour sans coupure — et revenir en arrière si ça tourne mal.

⌘ commandes Kubernetes · 🎓 quiz Kubernetes

  1. Vérifier le cluster et le contexte
    kubectl config current-context kubectl get nodes
  2. Créer un espace de noms dédié
    kubectl create namespace demo
  3. Déployer l'application (2 réplicas)
    kubectl create deployment web --image=nginx:1.27 --replicas=2 -n demo
  4. L'exposer sur le réseau
    kubectl expose deployment web --port=80 --type=NodePort -n demo kubectl get svc -n demo
  5. Surveiller les pods
    kubectl get pods -n demo -w kubectl logs -f deployment/web -n demo
  6. Monter en charge
    kubectl scale deployment web --replicas=4 -n demo
  7. Mettre à jour sans coupure… et savoir reculer
    kubectl set image deployment/web nginx=nginx:1.28 -n demo kubectl rollout status deployment/web -n demo kubectl rollout undo deployment/web -n demo

    Le rollout undo revient à la version précédente en quelques secondes.

  8. Tout nettoyer d'un coup
    kubectl delete namespace demo

Automatiser avec un premier playbook Ansible
Ansible · YAML — 7 étapes
▼

Configurer un serveur à la main, c'est bien une fois. Ansible rejoue la même configuration sur 1 ou 100 machines, sans rien installer sur elles (tout passe par SSH).

⌘ commandes Ansible · 🎓 quiz Ansible

  1. Installer Ansible — sur ton poste seulement
    sudo apt install ansible
  2. Décrire tes serveurs dans un inventaire
    nano hosts.ini 
    [web]
srv1 ansible_host=192.168.1.50 ansible_user=tom
  3. Tester la connexion
    ansible -i hosts.ini web -m ping

    Réponse attendue : "ping": "pong" — SSH par clé doit déjà fonctionner.

  4. Écrire le playbook
    nano site.yml 
    - name: Serveur web de base
  hosts: web
  become: true
  tasks:
    - name: Installer nginx
      apt:
        name: nginx
        state: present
        update_cache: true
    - name: Demarrer et activer nginx
      service:
        name: nginx
        state: started
        enabled: true
  5. Simuler avant d'appliquer
    ansible-playbook -i hosts.ini site.yml --check --diff

    --check montre ce qui changerait sans rien toucher.

  6. Appliquer pour de vrai
    ansible-playbook -i hosts.ini site.yml
  7. Vérifier l'idempotence
    ansible-playbook -i hosts.ini site.yml

    Relancé une 2e fois : changed=0 — tout était déjà en place. C'est ça, la magie d'Ansible.

Monter un VPN WireGuard
WireGuard · ufw — 7 étapes
▼

Un tunnel chiffré entre ton poste et ton serveur en quelques minutes : accéder à ton réseau distant comme si tu y étais, avec le VPN le plus simple du moment.

  1. Installer WireGuard — des deux côtés
    sudo apt install wireguard
  2. Générer une paire de clés — sur chaque machine
    wg genkey | tee private.key | wg pubkey > public.key

    La clé privée ne quitte JAMAIS sa machine ; seule la publique s'échange.

  3. Configurer le serveur
    sudo nano /etc/wireguard/wg0.conf 
    [Interface]
PrivateKey = <privée du serveur>
Address = 10.8.0.1/24
ListenPort = 51820

[Peer]
PublicKey = <publique du client>
AllowedIPs = 10.8.0.2/32
  4. Configurer le client
    [Interface]
PrivateKey = <privée du client>
Address = 10.8.0.2/24

[Peer]
PublicKey = <publique du serveur>
Endpoint = ip-du-serveur:51820
AllowedIPs = 10.8.0.0/24
PersistentKeepalive = 25

    AllowedIPs = 0.0.0.0/0 à la place ferait passer TOUT ton trafic par le VPN.

  5. Ouvrir le port sur le serveur
    sudo ufw allow 51820/udp
  6. Activer le tunnel — des deux côtés
    sudo systemctl enable --now wg-quick@wg0
  7. Vérifier que ça passe
    sudo wg show ping 10.8.0.1

    Un latest handshake récent dans wg show = le tunnel est monté.
