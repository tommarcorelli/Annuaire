Déployer un site Nginx avec HTTPS
Debian/Ubuntu · Nginx · Let's Encrypt — 7 étapes
▼

De zéro à un site servi en HTTPS avec un certificat gratuit qui se renouvelle tout seul. Prérequis : un serveur Debian/Ubuntu et un nom de domaine qui pointe déjà vers son IP.

⌘ commandes Nginx · 🎓 quiz Nginx

  1. Installer Nginx et l'activer au démarrage
    sudo apt update && sudo apt install nginx sudo systemctl enable --now nginx
  2. Vérifier qu'il répond
    curl -I http://localhost

    Tu dois voir HTTP/1.1 200 OK — la page d'accueil par défaut de Nginx.

  3. Créer la racine du site
    sudo mkdir -p /var/www/monsite echo '<h1>Ça marche !</h1>' | sudo tee /var/www/monsite/index.html
  4. Créer le bloc serveur (vhost)
    sudo nano /etc/nginx/sites-available/monsite 
    server {
    listen 80;
    server_name monsite.fr www.monsite.fr;
    root /var/www/monsite;
    index index.html;
}
  5. Activer le site et recharger
    sudo ln -s /etc/nginx/sites-available/monsite /etc/nginx/sites-enabled/ sudo nginx -t sudo systemctl reload nginx

    nginx -t valide la syntaxe AVANT de recharger — toujours le lancer d'abord.

  6. Ouvrir le pare-feu
    sudo ufw allow 'Nginx Full'

    Le profil « Nginx Full » ouvre les ports 80 (HTTP) et 443 (HTTPS).

  7. Passer en HTTPS avec Let's Encrypt
    sudo apt install certbot python3-certbot-nginx sudo certbot --nginx -d monsite.fr -d www.monsite.fr sudo certbot renew --dry-run

    Certbot modifie le vhost tout seul et installe le renouvellement automatique ; le --dry-run vérifie qu'il fonctionnera.

Sécuriser un serveur Linux fraîchement installé
Debian/Ubuntu · SSH · ufw · fail2ban — 7 étapes
▼

Les 20 premières minutes d'un serveur exposé sur Internet : compte admin, connexion par clé, SSH durci, pare-feu et bannissement automatique des attaquants.

⌘ commandes OpenSSL/SSH · 🎓 quiz OpenSSL/SSH

  1. Tout mettre à jour
    sudo apt update && sudo apt full-upgrade
  2. Créer un utilisateur admin (on ne travaille jamais en root)
    sudo adduser tom sudo usermod -aG sudo tom
  3. Générer une clé SSH — sur TON poste, pas sur le serveur
    ssh-keygen -t ed25519 -C "tom@poste" ssh-copy-id tom@ip-du-serveur

    Teste la connexion par clé (ssh tom@ip-du-serveur) avant l'étape suivante, sinon tu risques de t'enfermer dehors.

  4. Durcir la config SSH
    sudo nano /etc/ssh/sshd_config 
    PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
    sudo systemctl restart ssh
  5. Activer le pare-feu (SSH d'abord !)
    sudo ufw default deny incoming sudo ufw allow OpenSSH sudo ufw enable
  6. Bannir les attaques par force brute
    sudo apt install fail2ban sudo systemctl enable --now fail2ban sudo fail2ban-client status sshd
  7. Mises à jour de sécurité automatiques
    sudo apt install unattended-upgrades sudo dpkg-reconfigure -plow unattended-upgrades

Créer un VLAN sur un switch Cisco
Cisco IOS — 7 étapes
▼

Segmenter le réseau : créer un VLAN, y affecter des ports, monter un trunk vers un autre switch, puis sauvegarder pour que tout survive au redémarrage.

⌘ commandes Cisco IOS · 🎓 quiz Cisco IOS

  1. Passer en mode configuration
    enable configure terminal
  2. Créer le VLAN et le nommer
    vlan 10 name BUREAU exit
  3. Affecter un port au VLAN
    interface fastEthernet 0/1 switchport mode access switchport access vlan 10 exit
  4. Ou toute une plage de ports d'un coup
    interface range fastEthernet 0/2 - 12 switchport mode access switchport access vlan 10 exit
  5. Monter le trunk vers l'autre switch
    interface gigabitEthernet 0/1 switchport mode trunk switchport trunk allowed vlan 10,20 end

    Le trunk transporte plusieurs VLAN sur un seul lien ; allowed vlan limite lesquels.

  6. Vérifier
    show vlan brief show interfaces trunk
  7. Sauvegarder — sinon tout disparaît au reboot
    copy running-config startup-config

Créer puis sauvegarder une VM sur Proxmox
Proxmox VE · qm · vzdump — 7 étapes
▼

Tout en ligne de commande : créer une VM Debian, la démarrer, puis la sauvegarder à chaud et savoir la restaurer.

⌘ commandes Proxmox · 🎓 quiz Proxmox

  1. Repérer le stockage et les ISO disponibles
    pvesm status ls /var/lib/vz/template/iso
  2. Créer la VM (id 100)
    qm create 100 --name debian-vm --memory 2048 --cores 2 --net0 virtio,bridge=vmbr0
  3. Ajouter le disque et l'ISO d'installation
    qm set 100 --scsi0 local-lvm:32 qm set 100 --cdrom local:iso/debian-12.iso qm set 100 --boot order='scsi0;ide2'
  4. Démarrer et suivre
    qm start 100 qm status 100

    L'installation se termine dans la console de l'interface web (ou qm terminal 100 si la VM a un port série).

  5. Lister les VM du nœud
    qm list
  6. Sauvegarder à chaud (snapshot)
    vzdump 100 --storage local --mode snapshot --compress zstd

    Le mode snapshot sauvegarde sans arrêter la VM.

  7. Restaurer (vers un nouvel id 101)
    qmrestore /var/lib/vz/dump/vzdump-qemu-100-*.vma.zst 101

Conteneuriser et publier une application avec Docker
Docker · Docker Hub — 7 étapes
▼

Du code source à l'image publiée sur Docker Hub : construire, lancer, déboguer, publier, nettoyer.

⌘ commandes Docker · 🎓 quiz Docker

  1. Écrire le Dockerfile à la racine du projet
    FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]
  2. Construire l'image
    docker build -t mon-app:1.0 .
  3. Lancer un conteneur
    docker run -d -p 8080:3000 --name app mon-app:1.0

    -p 8080:3000 : le port 3000 du conteneur devient le 8080 de ta machine.

  4. Vérifier que ça tourne
    docker ps docker logs -f app curl http://localhost:8080
  5. Entrer dans le conteneur pour déboguer
    docker exec -it app sh
  6. Publier sur Docker Hub
    docker login docker tag mon-app:1.0 monuser/mon-app:1.0 docker push monuser/mon-app:1.0
  7. Nettoyer
    docker stop app && docker rm app docker image prune

Installer MySQL et automatiser les sauvegardes
MySQL/MariaDB · cron — 6 étapes
▼

Base de données prête pour une appli : installation sécurisée, utilisateur dédié aux droits limités, et un dump compressé chaque nuit à 3 h.

⌘ commandes MySQL · 🎓 quiz MySQL

  1. Installer et sécuriser
    sudo apt install mysql-server sudo mysql_secure_installation
  2. Créer la base et l'utilisateur applicatif
    sudo mysql 
    CREATE DATABASE boutique;
CREATE USER 'app'@'localhost' IDENTIFIED BY 'MotDePasseFort!42';
GRANT ALL PRIVILEGES ON boutique.* TO 'app'@'localhost';
FLUSH PRIVILEGES;

    L'appli n'a de droits QUE sur sa base — jamais root dans le code.

  3. Tester la connexion
    mysql -u app -p boutique
  4. Faire un dump manuel
    mysqldump -u root -p boutique > boutique-$(date +%F).sql
  5. Automatiser chaque nuit à 3 h avec cron
    sudo crontab -e 
    0 3 * * * mysqldump boutique | gzip > /var/backups/boutique-$(date +\%F).sql.gz

    Dans un crontab, le % doit être échappé en \%. Pense à créer /var/backups et à donner l'accès root via ~/.my.cnf.

  6. Restaurer un dump
    gunzip < /var/backups/boutique-2026-07-03.sql.gz | mysql -u root -p boutique

Diagnostiquer une panne réseau, étape par étape
Linux · ip · dig · tcpdump — 7 étapes
▼

« Internet ne marche plus » : la méthode en couches, de la carte réseau jusqu'à la capture de paquets, pour localiser la panne au lieu de deviner.

⌘ commandes universelles · ⌘ tcpdump · 🎓 quiz

  1. La machine a-t-elle une IP et une route ?
    ip a ip route

    Pas d'IP → problème DHCP/câble. Pas de default via … → pas de passerelle.

  2. La passerelle répond-elle ?
    ping -c 4 192.168.1.1

    KO → problème local (câble, Wi-Fi, box). OK → on continue vers l'extérieur.

  3. Internet répond-il en IP pure ?
    ping -c 4 8.8.8.8

    Si l'IP répond mais pas les sites web → c'est le DNS, pas la connexion.

  4. Le DNS résout-il ?
    dig monsite.fr resolvectl status
  5. Le service distant écoute-t-il ?
    nc -zv monsite.fr 443 ss -tulnp

    nc -zv teste un port distant ; ss -tulnp liste ce qui écoute en local.

  6. Où ça casse en chemin ?
    traceroute 8.8.8.8
  7. En dernier recours : voir les paquets
    sudo tcpdump -i eth0 -n host 8.8.8.8

    Si les paquets partent sans réponse, la panne est en amont — pas sur ta machine.

Déployer une application sur Kubernetes
Kubernetes · kubectl — 8 étapes
▼

Le cycle de vie complet d'une appli sur un cluster : déployer, exposer, monter en charge, mettre à jour sans coupure — et revenir en arrière si ça tourne mal.

⌘ commandes Kubernetes · 🎓 quiz Kubernetes

  1. Vérifier le cluster et le contexte
    kubectl config current-context kubectl get nodes
  2. Créer un espace de noms dédié
    kubectl create namespace demo
  3. Déployer l'application (2 réplicas)
    kubectl create deployment web --image=nginx:1.27 --replicas=2 -n demo
  4. L'exposer sur le réseau
    kubectl expose deployment web --port=80 --type=NodePort -n demo kubectl get svc -n demo
  5. Surveiller les pods
    kubectl get pods -n demo -w kubectl logs -f deployment/web -n demo
  6. Monter en charge
    kubectl scale deployment web --replicas=4 -n demo
  7. Mettre à jour sans coupure… et savoir reculer
    kubectl set image deployment/web nginx=nginx:1.28 -n demo kubectl rollout status deployment/web -n demo kubectl rollout undo deployment/web -n demo

    Le rollout undo revient à la version précédente en quelques secondes.

  8. Tout nettoyer d'un coup
    kubectl delete namespace demo

Automatiser avec un premier playbook Ansible
Ansible · YAML — 7 étapes
▼

Configurer un serveur à la main, c'est bien une fois. Ansible rejoue la même configuration sur 1 ou 100 machines, sans rien installer sur elles (tout passe par SSH).

⌘ commandes Ansible · 🎓 quiz Ansible

  1. Installer Ansible — sur ton poste seulement
    sudo apt install ansible
  2. Décrire tes serveurs dans un inventaire
    nano hosts.ini 
    [web]
srv1 ansible_host=192.168.1.50 ansible_user=tom
  3. Tester la connexion
    ansible -i hosts.ini web -m ping

    Réponse attendue : "ping": "pong" — SSH par clé doit déjà fonctionner.

  4. Écrire le playbook
    nano site.yml 
    - name: Serveur web de base
  hosts: web
  become: true
  tasks:
    - name: Installer nginx
      apt:
        name: nginx
        state: present
        update_cache: true
    - name: Demarrer et activer nginx
      service:
        name: nginx
        state: started
        enabled: true
  5. Simuler avant d'appliquer
    ansible-playbook -i hosts.ini site.yml --check --diff

    --check montre ce qui changerait sans rien toucher.

  6. Appliquer pour de vrai
    ansible-playbook -i hosts.ini site.yml
  7. Vérifier l'idempotence
    ansible-playbook -i hosts.ini site.yml

    Relancé une 2e fois : changed=0 — tout était déjà en place. C'est ça, la magie d'Ansible.

Monter un VPN WireGuard
WireGuard · ufw — 7 étapes
▼

Un tunnel chiffré entre ton poste et ton serveur en quelques minutes : accéder à ton réseau distant comme si tu y étais, avec le VPN le plus simple du moment.

  1. Installer WireGuard — des deux côtés
    sudo apt install wireguard
  2. Générer une paire de clés — sur chaque machine
    wg genkey | tee private.key | wg pubkey > public.key

    La clé privée ne quitte JAMAIS sa machine ; seule la publique s'échange.

  3. Configurer le serveur
    sudo nano /etc/wireguard/wg0.conf 
    [Interface]
PrivateKey = <privée du serveur>
Address = 10.8.0.1/24
ListenPort = 51820

[Peer]
PublicKey = <publique du client>
AllowedIPs = 10.8.0.2/32
  4. Configurer le client
    [Interface]
PrivateKey = <privée du client>
Address = 10.8.0.2/24

[Peer]
PublicKey = <publique du serveur>
Endpoint = ip-du-serveur:51820
AllowedIPs = 10.8.0.0/24
PersistentKeepalive = 25

    AllowedIPs = 0.0.0.0/0 à la place ferait passer TOUT ton trafic par le VPN.

  5. Ouvrir le port sur le serveur
    sudo ufw allow 51820/udp
  6. Activer le tunnel — des deux côtés
    sudo systemctl enable --now wg-quick@wg0
  7. Vérifier que ça passe
    sudo wg show ping 10.8.0.1

    Un latest handshake récent dans wg show = le tunnel est monté.

Disque plein : trouver et faire de la place
Linux · df · du · find — 7 étapes
▼

« No space left on device » : la méthode pour localiser ce qui mange le disque et nettoyer sans supprimer au hasard.

⌘ commandes universelles · 🎓 quiz

  1. Confirmer et localiser la partition pleine
    df -h

    Repère la ligne à 100% — c'est souvent / ou /var, pas tout le disque.

  2. Descendre dossier par dossier vers le coupable
    sudo du -xh / --max-depth=1 2>/dev/null | sort -rh | head -15 sudo du -sh /var/* | sort -rh | head

    -x reste sur la même partition (ne compte pas les autres montages).

  3. Chasser les gros fichiers isolés
    sudo find / -xdev -type f -size +500M -exec ls -lh {} + 2>/dev/null
  4. Purger les journaux systemd
    journalctl --disk-usage sudo journalctl --vacuum-size=200M
  5. Nettoyer le gestionnaire de paquets
    sudo apt clean sudo apt autoremove --purge
  6. Si Docker est installé, c'est souvent lui
    docker system df docker system prune -a

    prune -a supprime TOUTES les images non utilisées par un conteneur — vérifie avant.

  7. Vérifier le résultat… et le piège classique
    df -h sudo lsof +L1 | head

    Si l'espace ne revient pas : un fichier supprimé est encore tenu ouvert par un processus (lsof +L1 les liste) — redémarre le service concerné.

Analyser les logs après un incident
Linux · journalctl · grep · awk — 7 étapes
▼

Le site est tombé cette nuit : reconstituer ce qui s'est passé à partir des journaux, du plus large au plus précis.

⌘ commandes universelles · 🎓 quiz

  1. Vue d'ensemble : les erreurs depuis le dernier boot
    journalctl -b -p err

    -p err filtre par priorité : seulement erreurs et pire.

  2. Cibler la fenêtre de temps de l'incident
    journalctl --since "2026-07-03 02:00" --until "2026-07-03 03:00"
  3. Suivre le service suspect
    journalctl -u nginx --since "02:00" tail -f /var/log/nginx/error.log

    tail -f suit le fichier en direct — pratique pour voir si l'erreur revient.

  4. Chercher un motif précis
    grep -ri 'timeout' /var/log/nginx/ grep -c 'error' /var/log/nginx/error.log
  5. Quelles IP frappent le plus ?
    awk '{print $1}' /var/log/nginx/access.log | sort | uniq -c | sort -rn | head

    Le combo sort | uniq -c | sort -rn = le classement des occurrences, à connaître par cœur.

  6. Extraire les erreurs serveur (5xx)
    awk '$9 ~ /^5/ {print $9, $7}' /var/log/nginx/access.log | sort | uniq -c | sort -rn | head
  7. Naviguer confortablement dans un gros log
    less /var/log/syslog

    Dans less : /motif cherche, G va à la fin, F suit en direct comme tail -f.

La machine rame : trouver le coupable
Linux · top · ps · lsof — 7 étapes
▼

CPU, mémoire, disque ou réseau ? La démarche pour identifier le processus qui plombe la machine, puis s'en débarrasser proprement.

⌘ commandes universelles · 🎓 quiz

  1. Vue temps réel
    htop

    Pas de htop ? top fait l'affaire : Maj+P trie par CPU, Maj+M par mémoire.

  2. Photographier les plus gourmands
    ps aux --sort=-%cpu | head -10 ps aux --sort=-%mem | head -10
  3. Regarder ce qui sort sur le réseau
    ss -tunap | head -20
  4. Qui tient ce port ou ce fichier ?
    sudo lsof -i :8080 sudo lsof +D /var/log
  5. Le noyau a-t-il quelque chose à dire ?
    sudo dmesg | tail -20

    Cherche les Out of memory: Killed process — l'OOM killer laisse toujours une trace.

  6. Tuer proprement (puis fermement)
    kill -15 1234 kill -9 1234

    -15 (TERM) demande poliment au programme de quitter ; -9 (KILL) l'abat sans lui laisser sauvegarder. Toujours essayer -15 d'abord.

  7. Surveiller que ça ne revient pas
    watch -n 2 'ps aux --sort=-%cpu | head -5'

Automatiser ses sauvegardes avec rsync + cron
Linux · rsync · tar · cron — 6 étapes
▼

Une copie miroir chaque nuit, une archive datée chaque semaine, une rotation automatique — et surtout : un test de restauration.

⌘ commandes universelles · 🎓 quiz

  1. Répétition générale (rien n'est copié)
    rsync -avn --delete ~/Documents/ /mnt/backup/documents/

    -n = dry-run : montre ce qui serait copié/supprimé. --delete supprime côté backup ce qui n'existe plus côté source — d'où l'intérêt de simuler d'abord.

  2. Première vraie copie
    rsync -a --delete ~/Documents/ /mnt/backup/documents/

    Le slash final sur Documents/ compte : avec, on copie le contenu ; sans, le dossier lui-même.

  3. Une archive datée et compressée
    tar -czf /mnt/backup/docs-$(date +%F).tar.gz ~/Documents
  4. Planifier : miroir chaque nuit, archive le dimanche
    crontab -e 
    0 2 * * * rsync -a --delete /home/tom/Documents/ /mnt/backup/documents/
0 3 * * 0 tar -czf /mnt/backup/docs-$(date +\%F).tar.gz /home/tom/Documents
  5. Rotation : supprimer les archives de plus de 30 jours
    find /mnt/backup -name 'docs-*.tar.gz' -mtime +30 -delete
  6. Tester la restauration — l'étape que tout le monde saute
    mkdir /tmp/restore-test && rsync -a /mnt/backup/documents/ /tmp/restore-test/ diff -r ~/Documents /tmp/restore-test

    Une sauvegarde jamais restaurée n'est pas une sauvegarde : c'est un espoir.

Utilisateurs, groupes et permissions propres
Linux · adduser · chmod · chown — 6 étapes
▼

Un collègue rejoint le projet : lui créer un compte, un dossier d'équipe partagé, et comprendre enfin ce que veut dire rwxr-xr--.

⌘ commandes universelles · 🎓 quiz

  1. Créer l'utilisateur
    sudo adduser alice
  2. L'ajouter aux bons groupes
    sudo usermod -aG sudo,www-data alice groups alice

    Sans -a, usermod REMPLACE les groupes au lieu d'ajouter — l'erreur classique.

  3. Savoir lire les permissions
    ls -l /srv

    rwxr-xr-- = propriétaire : lire/écrire/exécuter · groupe : lire/exécuter · autres : lire. En octal : 754.

  4. Créer un dossier d'équipe partagé
    sudo mkdir -p /srv/partage sudo chown root:www-data /srv/partage
  5. Permissions de groupe + héritage
    sudo chmod 2775 /srv/partage

    Le 2 devant (setgid) : tout fichier créé dedans appartiendra automatiquement au groupe www-data.

  6. Verrouiller un fichier sensible
    sudo chown root:root /etc/app/secret.conf sudo chmod 600 /etc/app/secret.conf

    600 : seul le propriétaire lit et écrit. C'est le réglage attendu pour clés privées et fichiers de mots de passe.

Transférer des fichiers entre deux machines
Linux · scp · rsync · SSH — 6 étapes
▼

Du petit fichier à la grosse archive : la bonne méthode selon la situation, avec vérification d'intégrité à l'arrivée.

⌘ commandes OpenSSL/SSH · 🎓 quiz

  1. Un fichier, vite fait
    scp rapport.pdf tom@serveur:/home/tom/
  2. Un dossier entier, avec reprise sur coupure
    rsync -avP ~/photos/ tom@serveur:photos/

    -P = barre de progression + reprise : si ça coupe à 80%, on ne repart pas de zéro. Pour les gros volumes, préfère toujours rsync à scp.

  3. Dans l'autre sens (serveur → poste)
    scp tom@serveur:/var/log/app.log .
  4. Gros dossier : compresser à la volée à travers SSH
    tar -czf - mondossier/ | ssh tom@serveur 'tar -xzf - -C /destination'

    Rien n'est écrit en local : l'archive voyage compressée dans le tunnel SSH et se décompresse à l'arrivée.

  5. Partage express sur le réseau local
    python3 -m http.server 8000

    Le dossier courant devient http://ton-ip:8000 — parfait pour dépanner, à couper juste après (aucune authentification).

  6. Vérifier l'intégrité à l'arrivée
    openssl dgst -sha256 rapport.pdf

    Lance la même commande des deux côtés : si les empreintes diffèrent, le fichier est corrompu.

Transformer son script en service systemd
Linux · systemd · journalctl — 6 étapes
▼

Ton appli tourne dans un terminal et meurt quand tu fermes SSH ? Un fichier unit et systemd la démarre au boot, la relance si elle plante et collecte ses logs.

⌘ commandes Debian/Ubuntu · 🎓 quiz

  1. Écrire l'unité
    sudo nano /etc/systemd/system/monapp.service 
    [Unit]
Description=Mon application
After=network.target

[Service]
User=tom
WorkingDirectory=/home/tom/monapp
ExecStart=/usr/bin/node server.js
Restart=on-failure

[Install]
WantedBy=multi-user.target

    Restart=on-failure : systemd relance l'appli toute seule si elle plante.

  2. Faire relire la configuration à systemd
    sudo systemctl daemon-reload
  3. Démarrer et vérifier
    sudo systemctl start monapp systemctl status monapp

    Cherche active (running) en vert. Sinon, le message d'erreur est juste en dessous.

  4. Activer au démarrage de la machine
    sudo systemctl enable monapp
  5. Suivre ses logs
    journalctl -u monapp -f
  6. Au quotidien
    sudo systemctl restart monapp journalctl -u monapp --since today

Identifier, formater et monter un disque
Linux · lsblk · parted · fstab — 7 étapes
▼

Un disque USB ou un deuxième disque dur tout neuf : le rendre utilisable, puis le faire monter automatiquement à chaque démarrage.

⌘ commandes universelles · 🎓 quiz

  1. L'identifier SANS se tromper
    lsblk sudo dmesg | tail -10

    Se tromper de disque à l'étape 4 = tout perdre. Vérifie la taille et l'absence de partitions montées. Ici on suppose que c'est /dev/sdb.

  2. Voir ce qu'il contient déjà
    sudo parted /dev/sdb print blkid
  3. Créer la table de partitions et une partition
    sudo parted /dev/sdb mklabel gpt sudo parted -a optimal /dev/sdb mkpart primary ext4 0% 100%
  4. Formater en ext4
    sudo mkfs.ext4 -L DATA /dev/sdb1

    -L DATA donne un nom (label) — pratique pour s'y retrouver dans lsblk.

  5. Monter
    sudo mkdir -p /mnt/data sudo mount /dev/sdb1 /mnt/data df -h /mnt/data
  6. Montage automatique au boot
    blkid /dev/sdb1 sudo nano /etc/fstab 
    UUID=xxxx-xxxx  /mnt/data  ext4  defaults  0  2

Un seul serveur, plusieurs applis : reverse proxy Nginx
Nginx · proxy_pass — 6 étapes
▼

Une appli Node sur le port 3000, une API Python sur le 8001, un seul serveur et une seule IP : Nginx reçoit tout sur le port 80/443 et distribue selon le domaine.

⌘ commandes Nginx · 🎓 quiz Nginx

  1. Vérifier que les applis tournent en local
    curl http://localhost:3000 ss -tulnp | grep -E '3000|8001'
  2. Créer le vhost de la première appli
    sudo nano /etc/nginx/sites-available/app1 
    server {
    listen 80;
    server_name app1.monsite.fr;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}

    Les proxy_set_header transmettent le vrai domaine et la vraie IP du visiteur à l'appli — sinon elle croit que tout vient de 127.0.0.1.

  3. Même chose pour la deuxième, autre domaine
    server {
    listen 80;
    server_name api.monsite.fr;

    location / {
        proxy_pass http://127.0.0.1:8001;
        proxy_set_header Host $host;
    }
}
  4. Activer les deux et recharger
    sudo ln -s /etc/nginx/sites-available/app1 /etc/nginx/sites-enabled/ sudo nginx -t && sudo systemctl reload nginx
  5. HTTPS pour tout le monde d'un coup
    sudo certbot --nginx -d app1.monsite.fr -d api.monsite.fr
  6. Si un domaine renvoie 502 : lire les logs
    sudo tail -f /var/log/nginx/error.log

    502 Bad Gateway = Nginx va bien mais l'appli derrière ne répond pas — vérifie qu'elle tourne (étape 1).

Filtrer le trafic avec une ACL Cisco
Cisco IOS · access-list — 6 étapes
▼

Autoriser le web, bloquer le reste : écrire une liste de contrôle d'accès, l'appliquer au bon endroit, et surtout la tester sans se couper la main.

⌘ commandes Cisco IOS · 🎓 quiz Cisco IOS

  1. Passer en configuration
    enable configure terminal
  2. Écrire l'ACL étendue
    access-list 100 permit tcp 192.168.10.0 0.0.0.255 any eq 443
access-list 100 permit tcp 192.168.10.0 0.0.0.255 any eq 80
access-list 100 permit udp 192.168.10.0 0.0.0.255 any eq 53

    Un deny ip any any implicite termine TOUTE ACL : ce qui n'est pas explicitement permis est bloqué. Et 0.0.0.255 est un masque inversé (wildcard), pas un masque de sous-réseau.

    interface gigabitEthernet 0/1 ip access-group 100 in exit

    in = trafic qui ENTRE dans le routeur par cette interface. Se tromper de sens est l'erreur classique.

  4. Vérifier ce qui est réellement appliqué
    show running-config | section access-list
  5. Tester depuis un poste du VLAN
    ping 8.8.8.8

    Le ping doit ÉCHOUER (ICMP n'est pas dans l'ACL) mais le web doit passer — c'est exactement ce qu'on a écrit.

  6. Sauvegarder
    copy running-config startup-config

Chiffrer un disque avec LUKS
Linux · cryptsetup — 7 étapes
▼

Un disque externe volé ou perdu ne doit rien révéler : LUKS chiffre toute la partition, déverrouillable par phrase de passe. Standard sur Linux, intégré au noyau.

⌘ commandes universelles · 🎓 quiz

  1. Identifier le disque (et vérifier trois fois)
    lsblk

    L'étape suivante EFFACE tout le contenu de la partition choisie. On suppose /dev/sdb1.

  2. Chiffrer la partition
    sudo cryptsetup luksFormat /dev/sdb1

    Il faut taper YES en majuscules, puis choisir une phrase de passe longue — c'est elle qui protège tout.

  3. Déverrouiller (ouvrir le coffre)
    sudo cryptsetup open /dev/sdb1 coffre

    La version déchiffrée apparaît sous /dev/mapper/coffre — c'est elle qu'on formate et qu'on monte, jamais /dev/sdb1 directement.

  4. Formater l'intérieur du coffre
    sudo mkfs.ext4 -L COFFRE /dev/mapper/coffre
  5. Monter et utiliser normalement
    sudo mkdir -p /mnt/coffre sudo mount /dev/mapper/coffre /mnt/coffre df -h /mnt/coffre
  6. Refermer proprement (les données redeviennent illisibles)
    sudo umount /mnt/coffre sudo cryptsetup close coffre
  7. Sauvegarder l'en-tête LUKS — ta ceinture de sécurité
    sudo cryptsetup luksHeaderBackup /dev/sdb1 --header-backup-file luks-header-sdb1.img

    Si l'en-tête est corrompu, les données sont perdues MÊME avec la phrase de passe. Ce fichier (à stocker ailleurs que sur le disque !) permet de le restaurer.
